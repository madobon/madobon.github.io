# Markdownブログの見た目を整えつつ、やりすぎた装飾を引き算した

ブログの Markdown 記事が少しシンプルすぎる気がして、記事詳細ページの見た目をまとめて見直しました。

最初は「ただの HTML 流し込み」感を減らしたかったのですが、途中で装飾を足しすぎる場面もあり、最終的には引き算もしながら整える流れになりました。

## 最初の依頼

出発点は、「Markdown のブログデザインがシンプルすぎないか」という確認でした。

一覧ページではカードの雰囲気が出ていた一方で、詳細ページの本文はかなり素直で、公開ノートとしては悪くないものの、読み物としての気分はもう少し作れそうでした。

そこで今回は `src/pages/BlogPostPage.vue` と `src/style.css` を中心に、記事詳細ページだけを少しエディトリアル寄りにする方針で触ることにしました。

## 実装でやったこと

まず `BlogPostPage.vue` 側では、本文の前に戻りリンクと公開日、タグをまとめたメタ帯を追加しました。

そのうえで本文は `blog-article-frame` という別の箱で包み、記事カードの中でも「読むための面」として見えるようにしています。

CSS 側では次のあたりをまとめて調整しました。

- 見出しの余白とアクセント線
- blockquote の背景と左境界線
- code block の質感
- Mermaid ブロックの囲い
- table や hr の見た目
- 記事カードの hover を止めて、読む面として安定させること

この時点では、Markdown 記事が以前よりかなり「ブログっぽい」見え方になりました。

## 途中で詰まったところ

ただ、見た目を盛ったぶん、和文記事との相性が微妙な部分もすぐ出てきました。

特に最初の段落の 1 文字だけ大きくする、いわゆる drop cap 風の表現は、英字ならともかく日本語ではかなり浮いて見えました。

実際に表示を確認すると、本文の出だしで視線が変に止まってしまい、自然に読み始められませんでした。ここは完全に「雰囲気は出るかもしれないけど、このサイトには合わない」パターンだったと思います。

もうひとつは本文幅です。

最初の実装では読みやすさを意識して少し細めに取っていたのですが、実際の画面では左右の余白が目立ってしまい、本文の密度より箱の大きさが気になる状態になっていました。

## 最終的な修正

最終的には、まず drop cap をやめました。

導入段落だけほんの少し文字サイズと行間を強めつつ、先頭 1 文字を特別扱いする装飾は外して、和文でも自然に読める状態に戻しています。

そのあと本文幅を段階的に広げて、最終的にはかなり余白を削る方向まで寄せました。

`src/style.css` の `blog-prose` は最大幅を広げ、`blog-article-frame` 側の左右パディングも少しずつ詰めています。
最終的には「読みやすさを壊さない範囲で、できるだけギリギリまで広げる」という調整になりました。

このあたりは最初から一発で決めるより、表示を見ながら少しずつ寄せたほうが早いと改めて感じました。

## まとめ

今回の変更で、ブログ詳細ページは以前よりだいぶ「読み物」としての空気が出せたと思います。

一方で、見た目を強くしすぎると和文の本文ではすぐ不自然になるので、足し算だけでなく、違和感が出た装飾をちゃんと引くのも大事でした。

実装後は `pnpm run build:site` でビルド確認をしていて、この記事追加後は `vp check` と `pnpm run build:site` も通しています。
